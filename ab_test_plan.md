# План A/B-тестирования: v1 vs v2

## 1. Гипотеза

**H₀ (нулевая):** F1-score модели v2 (GradientBoosting) на production-трафике не отличается от v1 (LogisticRegression).

**H₁ (альтернативная):** F1-score модели v2 статистически значимо выше, чем v1.

Уровень значимости: **α = 0.05**, мощность теста: **β = 0.20** (80 % шанс обнаружить реальное улучшение).

---

## 2. Разделение трафика

| Группа | Модель | Доля трафика | Эндпоинт |
|---|---|---|---|
| Контрольная (A) | v1 — LogisticRegression | 50 % | `/predict/v1` |
| Тестовая (B) | v2 — GradientBoosting | 50 % | `/predict/v2` |

**Механизм разделения:** при запросе к `/ab/predict` сервис вызывает `random.random() < 0.5` — независимое случайное распределение для каждого запроса (Bernoulli с p=0.5). Это гарантирует отсутствие систематического смещения при достаточном числе наблюдений.

**Стабильность группы для клиента:** при необходимости (кредитный скоринг — повторные обращения) группу фиксируют по `client_id % 2`:
```python
version = 'v1' if int(client_id) % 2 == 0 else 'v2'
```

## 3. Метрики успеха

### Основная метрика — F1-score класса «Дефолт»

F1 является гармоническим средним Precision и Recall:
```
F1 = 2 · (Precision · Recall) / (Precision + Recall)
```

**Почему F1?** В кредитном скоринге важны оба типа ошибок:
- **FP (False Positive)** → отказ платёжеспособному клиенту → потеря выручки
- **FN (False Negative)** → одобрение неплатёжеспособного клиента → убыток

F1 балансирует оба, не жертвуя одним ради другого.

### Дополнительная метрика — Precision

```
Precision = TP / (TP + FP)
```

**Почему Precision?** Банк несёт прямые финансовые потери при FP (одобрен кредит, который заканчивается дефолтом), поэтому важно понимать, насколько надёжны положительные предсказания модели. Recall выбирается как второстепенная метрика при регуляторных требованиях.

### Бизнес-метрики (см. ARCHITECTURE.md §5)

| Метрика | Единица | Цель |
|---|---|---|
| EPL (Expected Prevented Loss) | тыс. руб. | v2 ≥ v1 |
| ApprovalRate@Risk=5% | % | v2 ≥ v1 |

## 4. Статистический анализ

### Метод: z-тест для сравнения двух пропорций

В качестве прокси для F1 в онлайн-тесте используется **default_rate** (доля предсказанных дефолтов, которые оказались верными = Precision; или доля реальных дефолтов среди отправленных запросов).

```
z = (p̂₂ - p̂₁) / √(p̂(1-p̂)(1/n₁ + 1/n₂))
```

Где:
- `p̂₁`, `p̂₂` — наблюдаемые Precision в группах A и B
- `p̂ = (X₁ + X₂) / (n₁ + n₂)` — объединённая оценка
- `n₁`, `n₂` — размеры выборок

### Расчёт минимального размера выборки

Для ожидаемого эффекта Δ = 0.03 (улучшение F1 с 0.52 до 0.55):

```python
from statsmodels.stats.proportion import proportion_effectsize, zt_ind_solve_power

effect = proportion_effectsize(0.55, 0.52)   # Cohen's h
n = zt_ind_solve_power(effect_size=effect, alpha=0.05, power=0.80, alternative='larger')
# n ≈ 900 наблюдений на группу → 1800 total
```

### Доверительные интервалы (95 %)

```python
import numpy as np
from scipy import stats

def ci_proportion(successes, n, confidence=0.95):
    p_hat = successes / n
    z = stats.norm.ppf((1 + confidence) / 2)
    margin = z * np.sqrt(p_hat * (1 - p_hat) / n)
    return (p_hat - margin, p_hat + margin)
```

### Пример онлайн-расчёта через `/ab/stats`

```bash
curl http://localhost:5000/ab/stats
# {
#   "ab_stats": {
#     "v1": {"requests": 952, "default_rate": 0.2268, "avg_probability": 0.2041},
#     "v2": {"requests": 948, "default_rate": 0.1951, "avg_probability": 0.1734}
#   }
# }
```

```python
from scipy import stats
import numpy as np

n1, p1 = 952, 0.2268   # v1
n2, p2 = 948, 0.1951   # v2

# z-test for difference in proportions (H1: p2 < p1, меньше дефолтов у v2)
p_pool = (n1 * p1 + n2 * p2) / (n1 + n2)
z = (p2 - p1) / np.sqrt(p_pool * (1 - p_pool) * (1/n1 + 1/n2))
p_value = stats.norm.cdf(z)   # односторонний тест
print(f"z={z:.3f}, p-value={p_value:.4f}")
```

## 5. Продолжительность теста

| Параметр | Значение | Обоснование |
|---|---|---|
| Минимальный срок | **2 недели** | Захватывает недельную и двухнедельную сезонность |
| Максимальный срок | **4 недели** | Снижение риска дрейфа данных |
| Остановка досрочно | При p-value < 0.01 и n ≥ 1800 | Явный победитель / явный проигрыш |

**Внимание к peeking problem**: промежуточные проверки раз в сутки допустимы, но решение об остановке — только после достижения плановой мощности.

---

## 6. Критерий успешности

Модель v2 признаётся **победителем**, если выполнены **все** условия:

1. `p-value < 0.05` по z-тесту
2. Наблюдаемый `F1_v2 > F1_v1` на накопленных данных
3. `EPL_v2 ≥ EPL_v1` (бизнес-метрика не ухудшилась)
4. Нет статистически значимого роста latency (t-test на времена ответа)

Если хотя бы одно условие не выполнено — откат на v1.

## 7. Связь с архитектурой сервиса

| Механизм A/B | Реализация |
|---|---|
| Разделение трафика | `/ab/predict` → случайный выбор в `ABTestingManager` |
| Логирование группы | Поле `ab_group` в JSON-логе каждого запроса |
| Накопление статистики | In-memory счётчики в `ABTestingManager` (демо) / Redis в prod |
| Мониторинг | `GET /ab/stats` → Grafana dashboard |
| Переключение версий | Изменить переменную окружения `MODEL_DEFAULT_VERSION` без деплоя |

В production A/B-решение выносится на уровень балансировщика (nginx upstream weight / Istio traffic split), чтобы маршрутизация не зависела от Python-процесса.
